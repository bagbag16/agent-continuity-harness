# 检查清单 v2

## 防跑偏检查
- 当前输出是否仍在解决同一个核心问题
- 是否悄悄把任务换成了更容易回答的问题
- 是否引入了用户未要求的新主轴

## 信息状态检查
- 是否把 `暂定假设` 写成了 `已确认`
- 是否把 `待确认` 内容继续扩写成既定前提
- 是否清楚标记了哪些内容还没确认
- 是否只做了分类格式，却没有给出高影响分类的判断依据

## 语义保真检查
- 是否把 agent 的理解直接冒充成用户原意
- 是否在高影响改写时同时区分了原始表达和建议版本
- 是否在未确认前就把改写版写入沉淀内容

## 外化检查
- 后续关键判断是否依赖了未留档的重要信息
- 是否把“对话里提过一次”误当成“长期有效状态”
- 当前是否需要最小状态层来承接后续工作
- 是否把低影响内容也过度外化了
- 当前状态层是否过空，导致后续仍无法稳定承接
- 当前状态层是否过满，已经膨胀成任务管理清单
- 当前是否其实应建议转入 `cca`，而不是继续假设轻量承接足够
- 是否存在用户未明确提出、但高频出现或明显影响后续判断的必要内容，应先作为候选提示
- 外化候选是否说清了必要性、失败风险、归属对象和当前确认状态

## 一致性检查
- 前后定义是否一致
- 命名是否漂移
- 判断标准是否变化但未说明
- 新变化是否冲击了旧结论
- 是否只改局部没补整体

## 分层检查
- 当前轮是否混入了下一层决策
- 是否把问题定义、方案选择、执行细节、表达润色混在一起
- 是否为了省事跳过了本该先做的上一层澄清

## 推进检查
- 当前步是否已经足够清楚，可以交付最小结果
- 是否因为过度求稳而拖延了本可推进的当前步
- 暂停是否带有明确原因和恢复条件
- 是否把本应保持轻量的任务误触发成展开模式

## 主动纠偏检查
- 是否为了顺从用户路径，压掉了已发现的漏洞、约束冲突或核心漂移风险
- 是否存在小成本、低冲突、明显更优的实现路径却没有提示
- 是否把 agent 的更优路径误写成用户已确认目标
- 是否先点明“漏洞 / 影响 / 更优做法 / 是否需确认”，再继续执行
- 是否把本应升级到 `cca` 的状态、边界或归属问题继续留在轻量模式硬撑

## 技能防污染检查
- 本轮内容是否在悄悄改写 skill
- 是否把候选修改写成了既有规则
- 是否在没有用户明确要求时推进了 skill 更新

## 交付前检查
- 最终结果是否仍满足最初目标
- 是否违反了已确认约束
- 是否增加了新的冲突
- 是否把本该 agent 完成的整理工作重新丢给了用户
