# 路由

当当前决策依赖 `guard-mode` 与 `continuity-mode` 的切换，或需要记录路由切换原因时，读取本文件。

## 公开名与内部名

Agent Continuity Harness 是正式项目名。

`ach` 是调用缩写。

`adg` 与 `cca` 是内部实现概念：

- `adg` 提供 `guard-mode`
- `cca` 提供 `continuity-mode`

这种命名关系不是冲突。只有当两个文件对同一决策定义了不同权威行为时，才构成冲突。

## 默认路由

当任务可由轻量漂移控制、边界稳定和当前窗口读回处理时，默认留在 `guard-mode`。

只有当任务需要正式状态、恢复、交接、跨窗口延续或状态根治理时，才进入 `continuity-mode`。

## 升级到 `continuity-mode`

以下任一情况成立时升级：

- 后续工作会依赖当前目标、约束、决策或风险
- 任务跨越语义阶段边界
- 用户计划在另一个窗口或新对话继续
- 当前上下文已被证据、日志、网页、代码或 transcript 挤占
- 新信息改变或替代仍影响后续判断的旧决策
- 当前主问题是状态绑定、恢复、迁移或交接质量
- 审查或复核需要薄包，而不是继承完整 transcript

不要把固定对话轮数作为主要触发器。轮数只能提醒 agent 检查 continuity，真正触发由语义状态决定。

## 回到 `guard-mode`

只有以下条件成立时才回退：

- 正式状态已经足以支撑后续恢复
- 当前任务不再需要状态根操作
- 活跃问题重新回到局部收束或普通执行
- 若未来工作依赖本次切换，已有路由记录或 checkpoint

不得只为了降低流程感就回退；若未解决状态仍控制后续判断，应继续留在 `continuity-mode`。

## 路由切换记录

每次有意义切换都应记录：

- `direction`
- `reason`
- `task`
- `stage`
- `state_basis`
- `next_mode`

若已有正式状态根，记录应进入相关状态文件或 `state-manifest.json`，而不是只留在聊天里。

## 面向用户的规则

不要让普通用户选择 `adg` 还是 `cca`。

只有当当前模式影响下一步动作、状态写入、恢复或成本时，才解释模式。
